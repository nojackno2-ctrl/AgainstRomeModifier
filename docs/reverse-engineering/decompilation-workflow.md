This project keeps reverse-engineering output reproducible instead of treating
one conversation as the source of truth.

## Full EXE Inventory

The current function inventory is generated by:

```powershell
$root = 'C:\Users\nojac\AppData\Local\Temp\AgainstRome_RE'
$out = 'C:\Users\nojac\OneDrive\程式\Against_Rome_Modifier\re_workspace\ghidra_inventory'
$env:JAVA_HOME = Join-Path $root 'jdk21'
$env:PATH = (Join-Path $env:JAVA_HOME 'bin') + ';' + $env:PATH
& (Join-Path $root 'ghidra\support\analyzeHeadless.bat') `
  $root 'AgainstRomeVillageBuildArea' `
  -process 'Against_Rome.exe' `
  -scriptPath (Join-Path $root 'scripts') `
  -postScript 'GhidraFunctionInventory.java' $out decompile
```

Current generated artifacts:

- `re_workspace/ghidra_inventory/against_rome_function_index.csv`
  - 7381 functions plus names, sizes, parameter counts, call-reference counts,

- `re_workspace/ghidra_inventory/against_rome_decompiled_functions.c`
  - Ghidra pseudocode for the same 7381 functions.

These files are intentionally under `re_workspace/`, which is ignored by Git.
They are local evidence and lookup material. Do not publish original game files
or generated decompiled pseudocode.

## Evidence Rules

- Treat a Ghidra pseudocode function as a starting point, not original source.
- A file field is only "confirmed" when the EXE or script runtime path shows how
  the game consumes it.
- Unknown `FUN_*` functions remain unknown until they have call-path evidence,
  string evidence, or runtime behavior evidence.
- Modifier code should only expose a write patch after the expected bytes,
  file format, rollback path, and restore behavior are known.

## Reuse Rule

For future changes, start with:

1. `docs/reverse-engineering/`
2. `data/game_schema.json`
3. `re_workspace/ghidra_inventory/against_rome_function_index.csv`
4. `re_workspace/ghidra_inventory/against_rome_decompiled_functions.c`
5. A focused Ghidra script under `tools/re/` only if the existing evidence is
   not enough.
